# RV1126B factory_test 架构说明文档

## 1. 项目概述
`factory_test` 是运行在 **RV1126B** 平台上的设备工厂测试进程，通过 MQTT 协议与服务器/上位机通信，实现测试指令接收、分发执行与结果上报的完整闭环流程，同时支持 ADB TCP 方式的本地调试。

---

## 2. 整体架构概览
进程采用**多线程+分层解耦**的设计思想，核心分为三大线程模块与一个硬件抽象层：

| 层级/模块 | 线程标识 | 核心职责 |
| :--- | :--- | :--- |
| MQTT 客户端层 | 线程A | 网络通信：订阅测试指令主题、接收指令、结果上报、断线自动重连 |
| 指令分发层 | 线程B | 数据解析与路由：JSON 解析、指令分发、结果打包上报 |
| 硬件抽象层 | - | 统一硬件接口：模块注册表、通用 handle 接口、模块生命周期管理 |
| 测试执行层 | 线程C | 测试工作流：执行各模块测试程序、反馈执行结果 |

---

## 3. 模块详细说明

### 3.1 线程A：MQTT Client（网络通信层）
负责进程与外部的网络通信，是测试指令的入口与结果上报的出口。
- **核心功能**：
  1.  **订阅 Topic**：订阅服务器下发测试指令的 MQTT 主题，监听测试任务。
  2.  **回调结果**：接收线程B通过 `结果上报FIFO` 写入的测试结果，通过 MQTT 协议上报给服务器。
  3.  **自动重连**：网络异常断开时自动执行重连逻辑，保证通信链路稳定性。
- **数据交互**：
  - 接收指令：MQTT Topic → 线程A → `接受FIFO` → 线程B
  - 上报结果：线程B → `结果上报FIFO` → 线程A → MQTT Topic/ADB TCP

---

### 3.2 线程B：分发器（指令分发层）
是整个进程的“中枢调度器”，负责指令解析、路由分发与结果打包。
- **核心功能**：
  1.  **cJSON 解析器**：解析线程A通过 `接受FIFO` 传入的 JSON 格式测试指令，提取测试模块ID、参数等信息。
  2.  **路由分发**：根据解析出的模块ID，调用硬件抽象层的 `统一handle接口`，将指令提交给对应模块。
  3.  **cJSON 打包（结果上报）**：接收线程C通过 `返回结果FIFO` 写入的测试结果，组装为标准 JSON 格式，写入 `结果上报FIFO` 供线程A上报。
- **数据交互**：
  - 指令下发：线程A → 接受FIFO → 线程B → 硬件抽象层（`submit` 接口）
  - 结果回收：线程C → 返回结果FIFO → 线程B → 结果上报FIFO → 线程A

---

### 3.3 硬件抽象层（HAL）
为上层分发器与下层测试模块提供解耦的统一接口，屏蔽不同硬件模块的实现差异。
- **核心功能**：
  1.  **模块注册表**：维护所有支持测试的硬件模块信息（模块ID、测试入口、参数校验规则等），供分发器路由使用。
  2.  **统一handle接口**：提供统一的 `submit` 调用接口，分发器只需传入模块ID与参数，无需关心底层实现。
  3.  **生命周期管理**：管理测试模块的初始化、执行、销毁流程，测试结束后通知线程C结束工作流并解绑模块资源。

---

### 3.4 线程C：模块工作流（测试执行层）
负责具体硬件模块测试程序的执行，是测试逻辑的载体。
- **核心功能**：
  1.  **执行对应模块测试程序**：接收硬件抽象层下发的指令，调用对应模块的测试程序，执行工厂测试逻辑。
  2.  **结果反馈**：测试执行完成后，将测试结果（成功/失败、日志、数据）写入 `返回结果FIFO`，传递给线程B。
  3.  **资源回收**：接收硬件抽象层的结束通知，安全退出测试工作流，释放模块资源。

---

## 4. 核心数据流向
### 4.1 指令下发流程
```text
服务器/上位机 → MQTT Topic → 线程A(MQTT Client) → 接受FIFO → 线程B(分发器)
→ 解析JSON指令 → 路由到对应模块 → 硬件抽象层(统一handle接口 submit) → 线程C(模块工作流)
→ 执行测试程序
```

### 4.2 结果上报流程
```text
线程C(模块工作流) → 测试结果写入返回结果FIFO → 线程B(分发器)
→ JSON结果打包 → 写入结果上报FIFO → 线程A(MQTT Client) → MQTT Topic/ADB TCP → 服务器/上位机
```

---

## 5. 关键技术点
1.  **多线程同步**：通过 FIFO 实现线程间解耦通信，避免直接共享内存带来的竞态问题。
2.  **分层解耦设计**：网络层、分发层、抽象层、执行层职责清晰，模块新增/修改不影响其他层级。
3.  **模块可扩展**：新增测试模块仅需在硬件抽象层的模块注册表中注册，分发器无需修改即可路由。
4.  **链路稳定性**：MQTT 客户端支持断线自动重连，保证测试过程中网络异常的恢复能力。
5.  **本地调试支持**：支持通过 ADB TCP 连接 PC 上位机，实现本地测试与调试。

---

## 6. 扩展与维护说明
- **新增测试模块**：
  1.  在硬件抽象层实现模块测试程序，提供统一的调用入口。
  2.  在模块注册表中添加模块ID、入口地址、参数校验规则。
  3.  分发器与 MQTT 客户端无需修改，即可支持新模块的指令分发与结果上报。
- **问题定位**：
  - 网络问题：排查线程A的 MQTT 连接日志、FIFO 读写状态。
  - 指令解析问题：排查线程B的 JSON 解析日志、模块路由日志。
  - 测试执行问题：排查线程C的测试程序日志、硬件抽象层的模块生命周期日志。
